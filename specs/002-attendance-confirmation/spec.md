# Feature Specification: Attendance Confirmation

**Feature Branch**: `002-attendance-confirmation`
**Created**: 2026-03-14
**Status**: Draft

**Input**: Создатель завершённого события может подтвердить присутствие участников и опционально выставить каждому оценку 1–5. За оценку участник получает очки и уведомление в Telegram-бот.

## Clarifications

### Session 2026-03-14

- Q: Кто может подтверждать? → A: Только `creatorUserId` события.
- Q: Когда доступно подтверждение? → A: Только когда событие в статусе `past`.
- Q: Оценка обязательна? → A: Нет, опциональна. Без оценки — подтверждение без начисления очков.
- Q: Очки начисляются автоматически? → A: Да, `deltaPoints = rating` (1 оценка = 1 очко, 5 оценка = 5 очков). Заменяет существующий `attendance_feedback` (фиксированные 4 очка).
- Q: Видит ли участник свою оценку? → A: Нет, только получает начисленные очки.
- Q: Можно ли изменить оценку? → A: Нет, повторная отправка запрещена.
- Q: Формат запроса? → A: Один bulk-запрос списком `{ userId, rating? }`. Участники без подтверждения в списке игнорируются.
- Q: Нужны ли уведомления? → A: Да, через Telegram-бот с указанием оценки и количества начисленных очков.
- Q: UI-flow? → A: Организатор видит окно с оценками один раз после завершения события. После отправки окно больше не показывается.
- Q: Нужен эндпоинт для чтения подтверждений? → A: Нет. Флаг `attendanceConfirmed: boolean` добавляется в существующий `GET /events/:eventId`.

## User Scenarios & Testing

### User Story 1 — Bulk-подтверждение с оценками (Priority: P1)

Как создатель завершённого события, я могу одним запросом подтвердить присутствие участников и выставить каждому оценку, чтобы начислить им очки.

**Acceptance Scenarios**:

1. **Given** событие в статусе `past` и я его создатель, **When** отправляю `POST /events/:eventId/attendance` со списком участников и оценками, **Then** для каждого создаётся `AttendanceConfirmation`, начисляются очки равные оценке, отправляются уведомления.
2. **Given** участник в списке без оценки, **When** запрос выполнен, **Then** создаётся `AttendanceConfirmation` без начисления очков, уведомление отправляется без упоминания очков.
3. **Given** userId в списке не является участником события, **When** запрос выполнен, **Then** запись игнорируется без ошибки.
4. **Given** подтверждения уже были отправлены ранее, **When** попытка повторной отправки, **Then** возвращается `409 Conflict`.

### User Story 2 — Защита от неавторизованного доступа (Priority: P1)

**Acceptance Scenarios**:

1. **Given** я не создатель события, **When** отправляю запрос подтверждения, **Then** возвращается `403 Forbidden`.
2. **Given** событие в статусе `upcoming` или `ongoing`, **When** отправляю запрос, **Then** возвращается `400 Bad Request`.
3. **Given** событие `cancelled`, **When** отправляю запрос, **Then** возвращается `400 Bad Request`.

### User Story 3 — Флаг в деталях события (Priority: P2)

**Acceptance Scenarios**:

1. **Given** организатор ещё не подтвердил посещаемость, **When** запрашиваю `GET /events/:eventId`, **Then** `attendanceConfirmed: false`.
2. **Given** организатор уже отправил подтверждения, **When** запрашиваю `GET /events/:eventId`, **Then** `attendanceConfirmed: true`.

### Edge Cases

- Пустой список `attendances: []` — возвращать `400`.
- `rating` вне диапазона 1–5 — возвращать `400`.
- Ошибка отправки уведомления в Telegram не должна прерывать основной поток (fire-and-forget).
- Ошибка начисления очков одному участнику не должна откатывать подтверждения остальных (обрабатывать индивидуально).

## Requirements

### Functional Requirements

- **FR-001**: `POST /events/:eventId/attendance` доступен только `creatorUserId` события.
- **FR-002**: Запрос выполняется только если `computedStatus === past`. Для `cancelled`, `upcoming`, `ongoing` — `400`.
- **FR-003**: Тело запроса — массив `{ userId: string, rating?: 1|2|3|4|5 }`. Пустой массив — `400`.
- **FR-004**: Повторный вызов для уже подтверждённого события — `409 Conflict`.
- **FR-005**: Для каждого userId из списка: если пользователь является участником события — создаётся `AttendanceConfirmation`; иначе — запись игнорируется.
- **FR-006**: Если `rating` указан — начислять `deltaPoints = rating` через `PointsService.award` с `ruleCode: 'attendance_feedback'`, `referenceId: eventId`.
- **FR-007**: Уведомление отправляется каждому подтверждённому участнику через Telegram-бот (fire-and-forget).
- **FR-008**: Уведомление с оценкой: `"⭐ Организатор подтвердил твоё присутствие на «{title}» и поставил оценку {rating}/5. Начислено {rating} очков!"`.
- **FR-009**: Уведомление без оценки: `"✅ Организатор подтвердил твоё присутствие на «{название}»."`.
- **FR-010**: `GET /events/:eventId` возвращает поле `attendanceConfirmed: boolean`.
- **FR-011**: Участник не видит свою оценку нигде в API.

## Success Criteria

- **SC-001**: 100% подтверждённых участников с оценкой получают `deltaPoints = rating` в `PointsLedger`.
- **SC-002**: Повторный вызов всегда возвращает `409`, без дублирования записей и очков.
- **SC-003**: Ошибка Telegram-уведомления не влияет на HTTP-ответ эндпоинта.
- **SC-004**: `attendanceConfirmed` корректно переключается с `false` на `true` после первого вызова.
